# 规则提取覆盖率分析

## 一、规则方法概述

本项目的规则提取方法（`extract_with_rules.py`）使用正则表达式从候选事件文本中提取结构化信息，不依赖LLM。主要提取对象包括：
- 投资者/股东名称
- 认缴金额
- 认缴股数
- 价格
- 日期

## 二、规则覆盖的章节标题

规则通过 `locate_sections.py` 定位以下章节：

| 关键词 | 匹配率 (8家) | 说明 |
|--------|-------------|------|
| 股本演变 | 4/8 | 最常见标题命名 |
| 历史沿革 | 2/8 | 部分公司使用此命名 |
| 发行人股本演变 | 2/8 | 更正式的标题 |
| 股本及股权结构 | 1/8 | 少数公司使用 |

**定位成功率**: 8/8公司至少定位到1个目标章节，其中6/8定位到明确的"股本演变"章节。

## 三、规则能稳定抽取的事件类型

### 3.1 增资事件（规则覆盖率：约60-70%）

| 匹配模式 | 稳定性 | 说明 |
|----------|--------|------|
| "XX以货币方式出资XX万元认购新增注册资本XX万元" | 高 | 标准IPO增资表述，正则稳定匹配 |
| "XX以XX万元增资" | 高 | 简洁表述模式 |
| "注册资本由XX万元增至XX万元" | 中 | 能定位事件但投资者信息可能分散在上下文 |
| "新增注册资本由XX认购" | 中 | 需要上下文推断完整信息 |

## 四、规则无法稳定抽取的场景

### 4.1 需要LLM或人工复核的事件

1. **股权转让事件**: 正则难以区分"转让"与"增资"的资金流向。转让不改变总股本，但正则可能将其误判为增资。
2. **整体变更**: "整体变更为股份有限公司"是一种特殊的股本变化，正则难以理解其含义。
3. **吸收合并**: 如黄山谷捷吸收合并昆山谷捷，需理解事件性质而非简单匹配数字。
4. **零对价转让**: 金额为0的转让事件，正则可能因金额为空而漏过。
5. **代持还原**: 星图测控的代持还原场景，正则无法理解法律含义。
6. **设立出资**: 公司设立时的出资信息，格式多样，无统一模板。
7. **股权激励**: 以员工持股平台为媒介的股权激励，涉及复杂安排。

### 4.2 规则误抽的内容

| 误抽类型 | 原因 | 过滤方法 |
|----------|------|----------|
| 将股权转让误判为增资 | 两者关键词重叠（"出资"、"认购"） | LLM语义判断 |
| 将整体变更误判为普通增资 | 数字匹配成功但事件类型错误 | 关键词前置过滤 |
| 将存量表数字误抽为流量事件 | 表格中包含相似数字模式 | 上下文窗口扩大 |
| 重复抽取同一事件 | 同一事件在多处引用 | 去重算法 |

## 六、规则方法的优势和局限

### 优势
- **运行速度快**: 8家公司全量提取 < 10秒
- **零API成本**: 不需要调用LLM
- **结果可复现**: 相同输入始终产生相同输出
- **适合批量处理**: 可扩展到50+家公司

### 局限
- **无法理解语义**: 不能区分增资和股权转让
- **无法处理复杂事件**: 吸收合并、代持还原等
- **无法自动校验**: 不产生cross-check
- **依赖文本格式**: PDF解析质量影响极大
- **单位转换不可靠**: 不能自动识别并转换单位
- **无法发现遗漏**: 只能抽取匹配到的模式，不知道错过了什么

## 七、结论

规则提取方法适合作为流水线的**快速初筛层**：
- 承担章节定位和候选事件切块
- 提取格式规整的增资事件（约70-80%覆盖率）
- 将不确定事件推送到LLM层或人工复核队列

不建议规则单独作为最终提取方案，必须配合LLM做语义理解和分类，配合Pydantic做结构校验，配合cross-check做数值校验。
