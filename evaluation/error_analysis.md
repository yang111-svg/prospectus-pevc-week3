## Q5: 单位错误如何发现？

### 单位错误检测机制

1. **Pydantic Schema校验**: 检查数值字段格式（必须为数字）
2. **Cross-check流量存量校验**: `prev_total + subscription = next_total` 不一致时可能为单位错误
3. **价格反向验证**: `subscription_amount / subscription_qty = subscription_price`，若偏离合理范围则可能存在单位错误
4. **数量级检查**: 
   - 正常增资股数范围：数十到数千（万股）
   - 异常值：若 `subscription_qty_wan > 100000` 或 `< 0.1` 则标记

### 实际发现的单位问题

| 公司 | 问题 | 发现方式 |
|------|------|----------|
| 688758 赛分科技 | 原始数据曾将"股"填入`_wan`字段 | teacher feedback |
| 603418 友升股份 | 有限公司注册资本(万元)与股份公司股本(万股)混淆 | cross-check单位不匹配 |
| 301563 云汉芯城 | t0存量加总4599 vs 标注4598 | 逐股东核对 |

### 单位检测规则建议

```python
# 伪代码
if subscription_qty_wan > 50000:  # 5亿股异常
    flag("可能单位错误: 股未转换为万股")
if subscription_price > 1000:  # 1000元/股异常
    flag("可能单位错误: 价格异常")
if subscription_amount / subscription_qty != subscription_price:
    flag("金额/股数/价格不一致")
```

## Q6: 股权转让如何和增资认缴分开？

### 区分规则

| 判断维度 | 增资认缴 | 股权转让 |
|----------|----------|----------|
| 资金流向 | 投资人→公司 | 受让方→出让方 |
| 总股本变化 | 增加 | 不变 |
| 关键动词 | 增资、认购、出资、投入 | 转让、受让、出让、出售 |
| 涉及主体 | 公司+新/老股东 | 老股东+新股东（公司不参与） |
| 注册资本/股本 | 变化（增加） | 不变 |

### 实现方法

1. **关键词规则层**: 扫描文本关键词，初步分类
2. **LLM语义层**: LLM理解事件性质并标注event_type
3. **Cross-check校验层**: 
   - 若事件被标注为"增资"但 `prev_total == next_total` → 标记为疑似转让
   - 若事件被标注为"转让"但 `prev_total != next_total` → 标记为疑似增资
4. **人工复核**: 混合事件（如688758赛分科技同时增资+转让）进入复核队列

### 8家公司中的典型混合场景

- **688758 赛分科技**: 复星惟盈既增资又受让老股，需拆分为两条记录
- **301581 黄山谷捷**: 吸收合并（增资200万元）+ 股权转让（昆山谷捷→黄山供销等），两个独立事件
- **920116 星图测控**: 代持还原（转让）+ 星图空间增资 + 股权激励（转让）+ 策星银河增资，4个独立事件

## Q7: Cross-check失败后，流程如何处理？

### Cross-check失败处理流程

```
Cross-check 失败
    │
    ├── 差异 < 1% 且非零对价事件
    │   └── 标记为 "precision_issue"
    │       原因：四舍五入累积误差
    │       处理：自动调整（记录调整日志）
    │
    ├── 差异 1-10%
    │   └── 标记为 "review_needed"
    │       原因：可能单位错误、遗漏记录
    │       处理：进入人工复核队列
    │
    ├── 差异 > 10% 或数据不完整
    │   └── 标记为 "fail"
    │       原因：重大遗漏或错误
    │       处理：触发LLM重新提取该事件
    │       │
    │       ├── LLM重试成功 → 更新结果
    │       └── LLM重试失败 → 进入人工复核队列
    │
    └── 差异为 N/A（数据不完整）
        └── 标记为 "incomplete"
            处理：尝试从其他来源补全数据
```

### 实际发现并处理的fail案例

| 案例 | 差异 | 根因 | 处理方式 |
|------|------|------|----------|
| 301563 云汉芯城：570≠770 | 200万元 | 疑似遗漏增资记录 | 进入MRQ_001人工复核 |
| 688775 影石创新：137.0187≠144.2303 | 7.2万元 | 投资者认缴数取整差异 | 进入MRQ_003人工复核 |
| 301563 云汉芯城：4599≠4598 | 1万元 | 四舍五入累积误差 | 标记为precision_issue |

## Q8: 如果让这个流程扩展到50家公司，最先会失败在哪里？

### 扩展性风险排序

| 优先级 | 失败模式 | 风险等级 | 影响范围 |
|--------|----------|----------|----------|
| **P0** | PDF解析质量不一致 | 极高 | 全局 |
| **P0** | 章节定位模式覆盖不足 | 高 | 全局 |
| **P1** | 事件类型多样性超出预期 | 高 | 部分公司 |
| **P1** | LLM API成本爆炸 | 中高 | 全局（预算） |
| **P1** | 单位不一致（有限公司vs股份公司） | 中高 | 部分公司 |
| **P2** | 人工复核队列爆满 | 中 | 全局（效率） |
| **P2** | 股权转让检测召回率下降 | 中 | 部分公司 |
| **P3** | 中文NER精度下降 | 低 | 全局 |

### 详细分析

#### P0-1: PDF解析质量不一致（最先失败的点）

- 当前8家PDF均为可文本选择的格式（非扫描件）
- 扩展到50家时，可能遇到：
  - 扫描版PDF（需要OCR）
  - 表格在PDF中渲染为图像
  - 加密或受保护的PDF
  - 不同证监会信息披露格式的PDF
- **预估影响**: 50家中5-10家（10-20%）PDF解析质量严重下降

#### P0-2: 章节定位模式覆盖不足

- 当前关键词列表基于8家观察
- 50家公司可能使用不同的章节命名：
  - "历史沿革及股本形成"
  - "发行人设立及股本演变"
  - "公司设立及历次股本变动情况"
  - "发行人基本情况"（股本信息嵌入其中）
- **预估影响**: 10-15家（20-30%）章节定位不准确

#### P1-3: 事件类型多样性

- 8家中已遇到：增资、转让、整体变更、吸收合并、代持还原、股权激励
- 50家可能新增：减资、回购、债转股、分立、并购重组、员工持股计划多轮变更
- **预估影响**: 新增5-8种需处理的复杂事件类型

#### P1-4: LLM API成本

- 当前8家：约200次LLM调用（每家公司25条候选×8家）
- 50家：约1250次LLM调用
- 若使用Variant B（1500 tokens/条）：约200万tokens输出
- 按GPT-4o当前价格：约$20-30（仅提取，不含重试）
- 若加上失败重试：成本可能翻倍

### 缓解策略

1. **PDF解析层**: 建立多后端fallback机制（pdfplumber → PyMuPDF → OCR）
2. **章节定位层**: 建立50+关键词的扩展列表，加入模糊匹配
3. **事件分类层**: 先规则初筛→LLM精细分类→规则校验的三层架构
4. **成本控制**: 先用Variant A（简洁版）做全量初筛，仅对困难案例使用Variant B
5. **复用机制**: 相似公司共用章节定位规则和事件分类模型

## 总结

### 关键发现

1. **规则+LLM混合架构是最优解**: 规则负责定位和初筛（快、低成本），LLM负责语义理解和分类（准、高成本）
2. **Cross-check是质量守门员**: 发现云汉芯城200万元遗漏和影石创新7万元差异，证明了cross-check的必要性
3. **单位问题是系统性风险**: 有限公司/股份公司、元/万元、股/万股的三重转换是最常见错误源
4. **股权转让检测是主要弱点**: 当前流水线对股权转让的检测率显著低于增资
5. **PDF解析质量是扩展性瓶颈**: 50家规模下PDF多样性将成为最先暴露的问题

### 行动项

| 优先级 | 行动 | 预期效果 |
|--------|------|----------|
| P0 | 建立PDF解析fallback机制 | 适应扫描版/加密PDF |
| P0 | 扩展章节关键词列表至50+模式 | 提升章节定位覆盖率 |
| P1 | 完善股权转让检测（LLM+规则联合） | 股权转让召回率提升至80%+ |
| P1 | 实现自动化cross-check重试循环 | 减少人工介入 |
| P2 | 建立单位自动检测和转换模块 | 减少单位错误 |
| P2 | 批量处理架构设计 | 支撑50家规模 |
